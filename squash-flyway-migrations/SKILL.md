# Squash Flyway Migrations

Consolidate multiple accumulated Flyway migration files into a single baseline migration. Reduces clutter when migration history has grown large (20+ files), contains cancel-out patterns (ADD/DROP COLUMN pairs), or when preparing for a major release where new users would run all migrations from scratch.

## When to use

- Migration count has grown large (20+) and the history is mostly noise
- Preparing for a major release where new users would run all migrations from scratch
- Migrations contain `ADD COLUMN` / `DROP COLUMN` pairs that cancel out
- The user says "squash migrations", "collapse migrations", "consolidate migrations", "combine migrations", or "baseline migrations"

## When NOT to use

- Fewer than ~15 migrations (not enough clutter to justify the risk)
- During active development where migrations are being added frequently
- When any database is in an intermediate state that hasn't applied all migrations

## Prerequisites to verify before starting

1. **All databases are at the latest migration version.** No database should be in an intermediate state. Check `flyway_schema_history` on each environment.
2. **The app is not mid-deployment.** Squashing is a coordinated change across code + all databases.
3. **A backup exists** of the production database schema. Take one before starting.
4. **Platform/framework migrations** (from JARs, external dependencies) typically cannot be squashed — only squash migrations you own.

## Process

### Step 1: Inventory and classify migrations

List all migration files in the project's `db/migration/` directory. Separate them into:
- **Ownable** — migration files in the project source tree (can be squashed)
- **External** — migration files inside dependency JARs (cannot be modified, must remain)

If all migrations are external (e.g., from a platform JAR), stop — there's nothing to squash.

### Step 2: Determine the squash boundary

Identify the version number boundary:
- If squashing **all** ownable migrations: replace them with a single file at the lowest version (e.g., `V1000__baseline.sql`).
- If squashing **some** migrations (e.g., everything up to V1005): the new baseline covers V1000–V1005, newer migrations (V1006+) stay as-is.

The squash version must be:
- Higher than any external/dependency migration versions
- Lower than any migrations being kept

### Step 3: Generate the consolidated schema

Two approaches, in order of preference:

**Option A — Schema dump (recommended for production databases):**
```bash
# PostgreSQL: dump only the schema (no data) from a database at the latest version
pg_dump --schema-only --no-owner --no-privileges -U user -h host -d dbname > baseline.sql

# MySQL equivalent:
mysqldump --no-data --routines --triggers dbname > baseline.sql
```

This captures the exact current state including all indexes, constraints, and sequences.

**Option B — Concatenate migration files (for simpler cases):**
Concatenate the SQL from all ownable migration files into one file. This works when:
- Migrations are purely additive (no ALTER TABLE DROP, no renames)
- There are no conflicting statements (e.g., CREATE INDEX IF NOT EXISTS is fine)

If migrations contain `ALTER TABLE ADD COLUMN` followed by later `ALTER TABLE DROP COLUMN` on the same column, Option A is required — the dump produces clean end-state.

### Step 4: Write the baseline migration

Create a single new migration file (e.g., `V1000__baseline_schema.sql`) containing the consolidated schema from Step 3.

Add a header comment explaining what was squashed:
```sql
-- Consolidated baseline — replaces V1000 through V1005.
-- Generated by schema dump on YYYY-MM-DD from database at version V1005.
-- Do not modify; create new migrations above V1000 for future changes.
```

If the project uses PostgreSQL with extensions (like `pgvector`), ensure `CREATE EXTENSION IF NOT EXISTS vector;` is included at the top — it won't be in the schema dump but is required for a fresh database.

### Step 5: Delete the old migration files

Remove all ownable migration files that were consolidated into the baseline.

**Do NOT delete external/dependency migration files** — they live in JARs and can't be touched.

### Step 6: Update flyway_schema_history on all existing databases

For each existing database (dev, staging, production):

```sql
-- Delete old migration records that were squashed
DELETE FROM flyway_schema_history WHERE version IN ('1000', '1001', '1002', '1003', '1004', '1005');

-- Insert the new baseline record
INSERT INTO flyway_schema_history (installed_rank, version, description, type, script, checksum, installed_by, installed_on, execution_time, success)
VALUES (
  <next available rank>,
  '1000',
  'baseline schema',
  'SQL',
  'V1000__baseline_schema.sql',
  NULL,
  current_user,
  now(),
  0,
  true
);
```

**Alternative: use `flyway repair`** — if the new baseline file has a matching checksum, Flyway repair can clean up the history automatically. But with a schema dump, the checksum won't match, so manual SQL is more reliable.

### Step 7: Update test code

Any test that manually runs migrations by filename (e.g., calling `runMigration(handle, "db/migration/V1000__ainotebook_schema.sql")`) must be updated to reference the new single baseline filename instead.

Tests that use real Flyway (`flyway:migrate`) don't need changes — they'll pick up the new file automatically.

### Step 8: Verify

1. **Fresh database test:** Create an empty database, run Flyway migrations. The baseline should create the full schema from scratch. All tables, indexes, constraints present.
2. **Existing database test:** Start the app against an existing database. Flyway should validate the baseline record and report "Schema is up to date. No migration necessary."
3. **Run the test suite:** All existing tests should pass.

## Common pitfalls

- **Extension dependencies:** If using pgvector, PostGIS, etc., the `CREATE EXTENSION` statement must come before any table that uses the extension's types. Add it manually at the top of the baseline — `pg_dump --schema-only` does NOT include `CREATE EXTENSION`.
- **Owner/privilege stripping:** Always use `--no-owner --no-privileges` with `pg_dump` so the baseline is portable across environments.
- **Sequence values:** `pg_dump --schema-only` includes sequence definitions but not their current values. For a baseline, this is correct — fresh databases start sequences from 1.
- **Platform migrations:** If a dependency JAR contains migrations V1–V16 and your project has V1000+, you can only squash V1000+. The JAR migrations are immutable.
- **Checksum mismatch warning:** If using `flyway repair` and the consolidated SQL doesn't match the old checksums, Flyway will warn. The manual SQL approach (Step 6) avoids this entirely.
